# Session: 2026-02-26b

## Goal
Deep research into video corruption artifacts for MP4/MOV and MXF formats.

## What Was Done
- Researched MP4/MOV file structure (atoms, stbl index tables, NAL units)
- Researched MXF file structure (KLV encoding, partitions, essence containers)
- Researched glitch art techniques (datamoshing, FFglitch, pixel sorting, channel shifting)
- Catalogued corruption types by controllability and playability risk
- Saved 3 research documents to `docs/research/`:
  - `video-corruption-mp4-mov.md` — MP4/MOV structure, NAL units, corruption safety rules
  - `video-corruption-mxf.md` — MXF/KLV structure, DNxHD/ProRes/JPEG2000 corruption
  - `glitch-art-techniques.md` — Datamoshing, FFglitch, pixel sort, 7 corruption categories

## Key Findings
- **MP4/MOV:** Never corrupt moov atom. Target mdat bitstream only. I-frame corruption cascades through entire GOP. Skip first ~20 bytes of NAL slice header.
- **MXF:** Intra-only codecs (DNxHD, ProRes, JPEG2000) mean no cascading — each frame's corruption is isolated. JPEG2000 wavelet artifacts look fundamentally different from DCT block artifacts.
- **Glitch art taxonomy:** 7 categories identified — Datamosh, Motion Vector, Frequency Domain, Pixel Operations, Byte-Level, Temporal, Spatial. Each with distinct controllability and playability profiles.
- **FFglitch** is the gold standard for precise bitstream manipulation (MVs, DCT, QP). Operates via JavaScript scripting.
- **Player tolerance:** VLC > mpv > FFplay >> QuickTime/AVFoundation. Design for VLC-level playability, test on stricter players.

## Decisions
- MXF support gives a fundamentally different corruption aesthetic (isolated/tile-like vs flowing/cascading)
- 7-category taxonomy proposed for app's corruption mode organization

## Next Steps
- Use research to refine corruption engine implementation
- Prioritize MP4 H.264 corruption first (widest audience)
- Consider MXF as Phase 2 (professional users)
- Evaluate embedding FFglitch vs building custom bitstream parser
